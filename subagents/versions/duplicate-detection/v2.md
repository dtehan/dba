### Phase 3 — Exact Row Duplicate Detection (MULTISET Tables Only)

**Scope Limitation:** This phase applies exclusively to MULTISET tables. SET tables automatically reject exact duplicate rows during insert operations.

**3a — High-Performance Hash-Based Detection:**

```sql
SELECT TOP 50
    HASHROW(*) AS RowHash,
    CAST(COUNT(*) AS BIGINT) AS DupCount
FROM {{databaseName}}.{TABLE_NAME}
GROUP BY 1
HAVING COUNT(*) > 1
ORDER BY DupCount DESC
```

**Technical Note:** HASHROW(*) computes a hash across all columns per row, enabling efficient AMP-local computation with GROUP BY operations on a single hash value.

**CRITICAL VALIDATION REQUIREMENT:** HASHROW(*) can generate hash collisions where different rows produce identical hash values. Always validate findings in Phase 3b. Report hash-based counts as "probable duplicates pending validation."

**3b — Row-Level Validation Protocol:**

For the top 2-3 hash values with highest DupCount from 3a, retrieve actual row data:

```sql
SELECT TOP 10 *
FROM {{databaseName}}.{TABLE_NAME}
WHERE HASHROW(*) = {hash_value_from_3a}
```

**Validation Logic:** If all columns are identical across rows, confirm as exact duplicates. If column values differ, classify as hash collision and exclude from duplicate count with collision notation.

**3c — Statistical Estimation for Large Tables:**

For tables exceeding 1M rows, implement sampling-based duplicate rate estimation:

```sql
SELECT
    CAST(COUNT(*) AS BIGINT) AS SampledRows,
    CAST(SUM(CASE WHEN rh_cnt > 1 THEN 1 ELSE 0 END) AS BIGINT) AS RowsInDupGroups
FROM (
    SELECT HASHROW(*) AS rh,
           COUNT(*) OVER (PARTITION BY HASHROW(*)) AS rh_cnt
    FROM {{databaseName}}.{TABLE_NAME}
    SAMPLE 10000
) dt
```

**Reporting Standard:** Label all sampling-based results as "estimated based on 10,000-row sample."

For tables under 1M rows, Phase 3a results provide sufficient precision without sampling.

### Phase 4 — Business Key Duplicate Analysis (Universal Application)

**Scope:** This phase applies to ALL tables, utilizing the key columns identified in Phase 2.

**4a — Duplicate Key Group Identification:**

```sql

    {key_col_1}, {key_col_2},

GROUP BY {key_col_1}, {key_col_2}
```

### Phase 5 — Fuzzy Near-Duplicate Pattern Recognition

**Applicability:** Tables containing CHAR or VARCHAR columns in business keys or frequently matched attributes.

**Column Selection Strategy:**
1. String columns within the inferred business key from Phase 2
2. Semantic pattern matching: `*_NAME`, `*_NM`, `*_DESC`, `*_ADDR`, `*_ADDRESS`
3. Analysis scope limitation: 3-5 columns per table to maintain tool execution efficiency

**5a — Normalization-Based Similarity Detection:**

```sql
SELECT TOP 50
    UPPER(TRIM({string_col})) AS NormalizedVal,
    CAST(COUNT(*) AS BIGINT) AS DupCount,
    CAST(COUNT(DISTINCT {string_col}) AS BIGINT) AS RawDistinct
FROM {{databaseName}}.{TABLE_NAME}
GROUP BY UPPER(TRIM({string_col}))
HAVING COUNT(*) > 1 AND COUNT(DISTINCT {string_col}) > 1
ORDER BY DupCount DESC
```

**Filtering Logic:** The `COUNT(DISTINCT {string_col}) > 1` clause ensures only groups where raw values genuinely differ are flagged, filtering out exact duplicates already identified in Phases 3-4.

**Large Table Optimization:** For tables exceeding 1M rows, append `SAMPLE 10000` after the table reference.

**5b — Phonetic Similarity Analysis (Name Columns Only):**

Apply SOUNDEX exclusively to columns with name semantics (`*_NAME`, `*_NM`, `FIRST_NAME`, `LAST_NAME`, `COMPANY_NAME`):

```sql
SELECT TOP 50
    SOUNDEX({name_col}) AS PhoneticCode,
    CAST(COUNT(*) AS BIGINT) AS DupCount,
    CAST(COUNT(DISTINCT {name_col}) AS BIGINT) AS RawDistinct
FROM {{databaseName}}.{TABLE_NAME}
GROUP BY SOUNDEX({name_col})
HAVING COUNT(*) > 1 AND COUNT(DISTINCT {name_col}) > 1
ORDER BY DupCount DESC
```

**5c — Fuzzy Group Sample Extraction:**

For top findings from 5a or 5b, demonstrate raw value variations that normalize to identical values:

```sql
SELECT TOP 20
    {string_col},
    UPPER(TRIM({string_col})) AS Normalized,
    CAST(COUNT(*) AS BIGINT) AS Cnt
FROM {{databaseName}}.{TABLE_NAME}
GROUP BY {string_col}
HAVING UPPER(TRIM({string_col})) IN (
    SELECT TOP 5 UPPER(TRIM({string_col}))
    FROM {{databaseName}}.{TABLE_NAME}
    GROUP BY UPPER(TRIM({string_col}))
    HAVING COUNT(DISTINCT {string_col}) > 1
    ORDER BY COUNT(*) DESC
)
ORDER BY Normalized, {string_col}
```

### Phase 6 — Precision Remediation SQL Generation

Generate production-ready SQL scripts for each duplicate category. **ALL remediation SQL is advisory only — never execute DDL or DML operations.**

**For Exact Row Duplicates (MULTISET Tables):**

```sql
-- Deduplicate {TABLE_NAME}: retain one row per duplicate group
-- Step 1: Create deduplication staging table
CREATE TABLE {{databaseName}}.{TABLE_NAME}_dedup AS (
    SELECT DISTINCT * FROM {{databaseName}}.{TABLE_NAME}
) WITH DATA;

-- Step 2: Verification - row count validation
SELECT 'original' AS src, CAST(COUNT(*) AS BIGINT) AS cnt FROM {{databaseName}}.{TABLE_NAME}
UNION ALL
SELECT 'deduped', CAST(COUNT(*) AS BIGINT) FROM {{databaseName}}.{TABLE_NAME}_dedup;

-- Step 3: After verification, execute table swap
RENAME TABLE {{databaseName}}.{TABLE_NAME} TO {TABLE_NAME}_backup;
RENAME TABLE {{databaseName}}.{TABLE_NAME}_dedup TO {TABLE_NAME};

-- Step 4: After application validation, remove backup
-- DROP TABLE {{databaseName}}.{TABLE_NAME}_backup;
```

**For Business Key Duplicates (Retention Strategy):**

```sql
-- Deduplicate by business key, preserving most recent record
CREATE TABLE {{databaseName}}.{TABLE_NAME}_dedup AS (
    SELECT * FROM {{databaseName}}.{TABLE_NAME}
    QUALIFY ROW_NUMBER() OVER (
        PARTITION BY {key_col_1}, {key_col_2}
        ORDER BY {timestamp_or_ordering_col} DESC
    ) = 1
) WITH DATA;
```

**Ordering Column Selection Logic:** Prioritize timestamp/date columns when available, otherwise utilize Primary Index or surrogate key. Document selection rationale.

**For Fuzzy Near-Duplicates (Standardization):**

```sql
-- Standardize {string_col}: normalize case and eliminate whitespace
UPDATE {{databaseName}}.{TABLE_NAME}
SET {string_col} = UPPER(TRIM({string_col}))
WHERE {string_col} <> UPPER(TRIM({string_col}));
```

## Comprehensive Analysis Methodology

{{#queryText}}
### Phase 1: Execution Plan Analysis
Execute `EXPLAIN {{queryText}}` to obtain the complete execution strategy. Systematically examine the plan for performance anti-patterns:

**Critical Red Flags**:
- **Cartesian Products**: Any indication of "product join" or uncontrolled cross-joins
- **Full AMP Scans**: "all-AMPs RETRIEVE" operations without index utilization or partition pruning
- **Excessive Data Movement**: "redistributed by hash" patterns indicating suboptimal join strategies
- **Broadcast Operations**: "duplicated on all AMPs" suggesting inefficient small-table handling
- **Partition Inefficiency**: "all partitions" accessed on PPI tables instead of targeted partition elimination
- **Row Estimation Anomalies**: Grossly inflated or deflated cardinality estimates
- **Spool Contention**: Intermediate spool space projections that approach system limits

### Phase 2: Historical Performance Correlation
Query DBQL views to establish actual execution patterns and resource consumption baselines:

```sql
SELECT TOP 20
    QueryID, StartTime, UserName,
    CAST(TotalIOCount AS BIGINT) AS TotalIO,
    CAST(AMPCPUTime AS DECIMAL(18,2)) AS AMPCPUTime,
    CAST(SpoolUsage AS BIGINT) AS SpoolUsage,
    CAST(NumResultRows AS BIGINT) AS ResultRows,
    CAST(TotalFirstRespTime AS DECIMAL(18,4)) AS ElapsedSec,
    NumSteps, ErrorCode
FROM DBC.QryLogV
WHERE UserName = USER
  AND StartTime >= DATE - 7
  AND AMPCPUTime > 0
ORDER BY AMPCPUTime DESC
```

**Contingency Protocol**: If DBQL access is restricted or unavailable, document the limitation and proceed with EXPLAIN-only analysis while noting the reduced diagnostic scope.

### Phase 3: Step-Level Performance Decomposition
For identified high-impact queries, perform granular step analysis to isolate bottlenecks:

```sql
SELECT TOP 50
    StepNum, StepName,
    CAST(MaxAMPCPUTime AS DECIMAL(18,4)) AS MaxAMPCPU,
    CAST(AvgAMPCPUTime AS DECIMAL(18,4)) AS AvgAMPCPU,
    CAST(MaxAMPIO AS BIGINT) AS MaxAMPIO,
    CAST(MaxSpoolUsage AS BIGINT) AS MaxSpool,
    CAST(EstResultRows AS BIGINT) AS EstRows,
    CAST(ActResultRows AS BIGINT) AS ActRows
FROM DBC.QryLogStepsV
WHERE QueryID = {IDENTIFIED_QUERY_ID}
ORDER BY StepNum
```

**Diagnostic Focus Areas**:
- **Cardinality Estimation Accuracy**: Identify steps where ActRows significantly diverges from EstRows (>10x variance indicates statistics issues)
- **AMP Skew Detection**: MaxAMPCPU >> AvgAMPCPU suggests data distribution problems or hot-spotting
- **Resource Bottlenecks**: Steps consuming disproportionate spool space or I/O resources

### Phase 4: Statistics Coverage and Currency Assessment
Systematically evaluate the statistical foundation supporting query optimization:

```sql
SELECT TOP 50
    TRIM(DatabaseName) AS DatabaseName,
    TRIM(TableName) AS TableName,
    TRIM(ColumnName) AS ColumnName,
    StatsType, LastCollectTimestamp, SampleSizePct,
    CAST(UniqueValueCount AS BIGINT) AS UniqueValues
FROM DBC.StatsV
WHERE DatabaseName = '{EXTRACTED_DATABASE_NAME}'
  AND TableName = '{EXTRACTED_TABLE_NAME}'
ORDER BY LastCollectTimestamp ASC
```

**Statistics Quality Indicators**:
- **Missing Coverage**: Join predicates, filter conditions, and GROUP BY columns lacking statistics
- **Staleness Threshold**: Statistics older than 30 days on tables with regular DML activity
- **Sample Adequacy**: SampleSizePct < 10% on large tables may produce unreliable cardinality estimates

### Phase 5: Partitioning Optimization Verification
For Partitioned Primary Index (PPI) tables, assess partition elimination effectiveness:

```sql
SELECT TOP 20
    TRIM(DatabaseName) AS DatabaseName,
    TRIM(TableName) AS TableName,
    TRIM(ColumnName) AS ColumnName,
    IndexType, ColumnPosition
FROM DBC.IndicesV
WHERE DatabaseName = '{TARGET_DATABASE}'
  AND TableName = '{TARGET_TABLE}'
  AND IndexType = 'Q'
ORDER BY ColumnPosition
```

Verify that query predicates align with partitioning columns to enable partition pruning.

## Performance Impact Classification Framework

Apply rigorous severity assessment to prioritize remediation efforts:

**CRITICAL** (Immediate Action Required):
- Cartesian product joins causing exponential resource consumption
- Full-table scans on tables exceeding 1 million rows without justification
- Spool exhaustion scenarios threatening system stability
- Queries consuming >50% of system resources during execution

**HIGH** (Priority Remediation):
- Missing statistics on primary join/filter columns
- Cardinality estimation errors exceeding 100x actual values
- Unnecessary data redistribution patterns
- Queries with >60 second elapsed times showing optimization potential

**MEDIUM** (Planned Optimization):
- Statistics currency issues (>30 days stale on volatile tables)
- Suboptimal join ordering with measurable impact
- PPI tables missing partition elimination opportunities
- Moderate resource consumption with clear improvement paths

**LOW** (Maintenance Tasks):
- Minor query inefficiencies with minimal system impact
- Cosmetic SQL improvements without significant performance gain
- Slightly outdated statistics on relatively static tables
